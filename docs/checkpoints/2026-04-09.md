## Important Runtime Rules

- `/serverXR` must stay owned by the cPanel Node.js App.
- Do not delete the `serverXR` mount directory inside the web root.
- The Passenger `.htaccess` inside the mount directory is required.
- Staging and production do not automatically share data or assets.
- Code deploy and content/data deploy are separate concerns.

## Required Server Env Shape

### Staging

- `APP_BASE_PATH=/serverXR`

## What Changed In This Recovery

- Production was moved back to the Passenger-backed `/serverXR` mount instead of the broken proxy state.
- Staging was repaired after a missing shared-schema path caused the backend to fail on startup.
- The backend now supports a configurable `SHARED_ROOT` so staging can load `shared-staging` without pretending to be production.
- The deploy env examples now document `SHARED_ROOT`.
- The prebuilt publish workflow now tolerates repos where `serverXR/package-lock.json` may or may not exist.
- Non-`main` spaces now keep their own route and metadata context instead of drifting back to `main`.
- The work lane now supports direct management actions for spaces and projects.

## Known Caveats

- Assets and scenes under `serverXR/data` are environment-specific. Promoting code does not automatically promote media/content.
- Older deploy docs and older workflow files still exist in the repo, but they are not the primary path anymore.
- If staging or production looks stale after a good GitHub run, the missing step is usually cPanel `Deploy HEAD Commit`, not another code change.
